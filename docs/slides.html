作業環境 ／ 対象システムの概要

1. はじめに ― 作業環境

ツール変更の経緯

  • Before Astah でUML作成
    → バイナリファイル、差分管理が困難
  • After PlantUML に移行
    → テキスト形式、GitHubで差分確認可能

PlantUML 採用の効果

  • 変更点（誰が・どの図・どこを）を追いやすい
  • Pull Request・差分確認との相性が良い
  • チーム開発に適したUML管理を実現
PlantUML コード例
@startuml
class Guest {
  - name : String
  - contact : String
}
Guest "1" -- "0..*" Reservation
@enduml
↑ テキストで管理することでgit diffが有効活用できる

2. ドメイン分析 ― クラス図

主要概念（8つ）

  • 利用者（Guest）
  • ホテル（Hotel）
  • 客室（Room）
  • 予約（Reservation）
  • フロント係（FrontStaff）
  • チェックイン（CheckIn）
  • チェックアウト（CheckOut）
  • 宿泊料金（Charge）

主な関係

  • 利用者 → 複数の予約
  • 予約 → 1つの客室
  • ホテル → 複数の客室・フロント係
  • チェックアウト → 宿泊料金

要求分析

ユースケース図 ／ ユースケース記述

3. 要求分析 ― ユースケース図

ユースケース一覧

  • UC1：部屋を予約する
  • UC2：チェックインする
  • UC3：チェックアウトする
  • UC4：予約内容を確認する
  • UC5：予約をキャンセルする

アクター

  • 利用者：UC1, UC2, UC4, UC5
  • フロント係：UC2, UC3
include関係
UC2（チェックイン）→ <<include>> → UC4
UC5（キャンセル）→ <<include>> → UC4
事前に予約内容を確認する共通処理を分離
図3：ユースケース図

3. 要求分析 ― ユースケース記述

UC1：部屋を予約する
利用者が宿泊日・客室タイプを指定 → 空室一覧を提示 → 客室選択・氏名・連絡先入力 → 予約番号を発行
例外：空室なし、直前競合
UC2：チェックインする
フロント係が予約番号を確認 → チェックイン処理
代替：氏名＋宿泊日で検索
UC3：チェックアウトする
宿泊料金計算 → 追加サービス料金加算 → 支払い → 予約状態更新
例外：支払い拒否 → 料金再確認
UC4：予約内容を確認する
対象予約の存在・内容を確認する共通処理（UC2・UC5からinclude）
UC5：予約をキャンセルする
予約内容確認 → キャンセル処理
例外：キャンセル不可状態
各ユースケースで
事前条件 / 事後条件 / 基本フロー / 代替フロー / 例外フローを整理
Section 4

アクティビティ図

予約 ／ チェックイン ／ チェックアウト

4. アクティビティ図 ― 予約（UC1）

スイムレーン

  • 利用者
  • システム

処理の流れ

  • 宿泊日・客室タイプを指定
  • 空室の有無を判定 （空室なし → 終了）
  • 空室一覧を提示 → 客室選択
  • 直前競合チェック
  • 問題なければ予約番号を発行
図4：アクティビティ図（予約）

4. アクティビティ図 ― チェックイン（UC2）

スイムレーン

  • 利用者
  • フロント係
  • システム

処理の流れ

  • 予約番号あり → そのまま検索
  • 予約番号なし → 氏名＋宿泊日で検索
  • 該当予約があれば予約状態をチェックイン済みに更新
  • 部屋番号を通知
図5：アクティビティ図（チェックイン）

4. アクティビティ図 ― チェックアウト（UC3）

スイムレーン

  • 利用者
  • フロント係
  • システム

処理の流れ

  • 料金計算
  • 追加サービス料金の加算
  • 支払い確認
  • 支払いに不同意 → 料金内容を再確認
  • 解決しない場合 → チェックアウト未完了
  • 支払い記録 → 予約状態をチェックアウト済みに更新
図6：アクティビティ図（チェックアウト）
Section 5

システム分析とBCE

Boundary ／ Control ／ Entity

5. システム分析 ― BCE分類

🖥 Boundary（境界）
アクターとシステムの接点
  • ReservationUI
  • InquiryUI
  • CheckInUI
  • CheckOutUI
  • CancellationUI
⚙️ Control（制御）
業務ロジック・処理フロー
  • ReservationControl
  • InquiryControl
  • CheckInControl
  • CheckOutControl
  • CancellationControl
📦 Entity（実体）
業務データの保持
  • Reservation
  • Room
  • Guest
  • Charge
  • RoomRate
  • ServiceUsage
画面（B）は直接データを操作しない。B → Control → Entity の責務分担で変更に強い設計を実現
Section 6

コラボレーション図

UC1〜UC5 のメッセージ交換

6. コラボレーション図 ― UC1：予約

メッセージの流れ

  1. 利用者 → ReservationUI：宿泊日・客室タイプを入力
  2. ReservationUI → ReservationControl：空室検索依頼
  3. ReservationControl → Room：空室確認
  4. 利用者 → ReservationUI：客室選択
  5. ReservationControl → Room：予約済みに更新
  6. ReservationControl：Guest と Reservation を生成
  7. 予約番号 → ReservationUI → 利用者へ表示
図7：コラボレーション図（UC1 予約）

6. コラボレーション図 ― UC2：チェックイン

メッセージの流れ

  1. フロント係 → CheckInUI：予約番号を入力
  2. CheckInUI → CheckInControl：処理依頼
  3. CheckInControl → InquiryControl：予約内容確認
    （UC4のinclude関係をControl間の呼び出しとして実現）
  4. CheckInControl → Reservation：チェックイン済みに更新
  5. CheckInControl → Room：使用中に更新
ユースケース図のinclude関係が
Control間の呼び出しとして設計に反映
図8：コラボレーション図（UC2 チェックイン）

6. コラボレーション図 ― UC3：チェックアウト

メッセージの流れ

  1. フロント係 → CheckOutUI：チェックアウト操作
  2. CheckOutUI → CheckOutControl：処理依頼
  3. CheckOutControl → Charge・RoomRate：料金計算
  4. CheckOutControl → ServiceUsage：追加サービス料金加算
  5. 支払い確認後、CheckOutControl → Reservation：チェックアウト済みに更新
  6. CheckOutControl → Room：空室に更新
図9：コラボレーション図（UC3 チェックアウト）

6. コラボレーション図 ― UC4：予約照会

位置付け

UC2（チェックイン）・UC5（キャンセル）から
include される共通処理。

メッセージの流れ

  1. 利用者 → InquiryUI：予約番号を入力
  2. InquiryUI → InquiryControl：照会依頼
  3. InquiryControl → Reservation：予約情報を取得
  4. 予約内容を InquiryUI 経由で表示
共通処理として分離することで
重複なく複数UCから再利用
図10：コラボレーション図（UC4 照会）

6. コラボレーション図 ― UC5：キャンセル

メッセージの流れ

  1. 利用者 → CancellationUI：キャンセル操作
  2. CancellationUI → CancellationControl：処理依頼
  3. CancellationControl → InquiryControl：予約内容確認
    （UC4のinclude関係）
  4. CancellationControl → Reservation：キャンセル済みに更新
  5. CancellationControl → Room：空室に戻す
UC2と同様にInquiryControlを呼び出すことで
要求の構造が設計に一貫して反映されている
図11：コラボレーション図（UC5 キャンセル）

7. システムクラス図
