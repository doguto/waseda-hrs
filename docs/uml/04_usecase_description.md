# ユースケース記述

PlantUMLは図のための言語であり、ユースケース記述は本来テキスト/表形式で記述するため、
本ファイルのみMarkdown表で作成する。対象は業務フロー上の中核機能。

各ステップは現在時制・能動句（「〜する」）で記述し、利用者・フロント係の入力の直後に
システムの応答を明記する。代替フロー（正常系の分岐）と例外フロー（異常終了）は分けて記述する。

## UC1: 部屋を予約する

| 項目 | 内容 |
|---|---|
| アクター | 利用者 |
| 概要 | 利用者が希望する客室を選び予約を確定し、予約番号を受け取る |
| 事前条件 | 利用者が予約対象のホテル・宿泊日・客室タイプを把握している |
| 事後条件 | 予約が「予約済み」状態で登録され、予約番号が発行されている |
| 基本フロー | 1. 利用者が宿泊日・客室タイプを指定する<br>2. システムが該当する空室一覧を提示する<br>3. 利用者が客室を選択する<br>4. システムが予約を登録する<br>5. システムが予約番号を発行する<br>6. システムが予約番号を利用者に提示する |
| 代替フロー | 3a. 利用者が候補の客室すべてに対し気に入らず再検索を要求する場合<br>　3a1. 利用者が宿泊日・客室タイプを変更する<br>　3a2. 基本フローの手順2に戻る |
| 例外フロー | 2a. システムが該当する空室を1件も検出できない場合<br>　2a1. システムが「空室なし」を利用者に通知する<br>　2a2. ユースケースを終了する |

## UC2: チェックインする

| 項目 | 内容 |
|---|---|
| アクター | 利用者、フロント係 |
| 概要 | 利用者が予約番号を提示し、フロント係がチェックイン処理を行い部屋番号を通知する |
| 事前条件 | 予約が「予約済み」状態で存在する |
| 事後条件 | 予約が「チェックイン済み」状態になり、部屋番号が利用者に通知されている |
| 基本フロー | 1. 利用者が予約番号を提示する<br>2. フロント係が予約内容を確認する（include UC4）<br>3. システムが該当予約の内容をフロント係に提示する<br>4. フロント係がチェックイン処理を実行する<br>5. システムが予約状態をチェックイン済みに更新する<br>6. システムが部屋番号をフロント係に提示する<br>7. フロント係が部屋番号を利用者に通知する |
| 代替フロー | 1a. 利用者が予約番号を提示できない場合<br>　1a1. フロント係が氏名・宿泊日で予約を検索する<br>　1a2. 基本フローの手順3に進む |
| 例外フロー | 3a. システムが該当する予約を検出できない場合<br>　3a1. システムが「予約なし」をフロント係に通知する<br>　3a2. フロント係が利用者に再確認を依頼する<br>　3a3. ユースケースを終了する |

## UC3: チェックアウトする

| 項目 | 内容 |
|---|---|
| アクター | 利用者、フロント係 |
| 概要 | 宿泊終了後、フロント係がチェックアウト処理を行い宿泊料金を確定し、利用者が支払う |
| 事前条件 | 予約が「チェックイン済み」状態である |
| 事後条件 | 予約が「チェックアウト済み」状態になり、宿泊料金が支払い済みとして記録される |
| 基本フロー | 1. 利用者がフロントでチェックアウトを申告する<br>2. システムが宿泊料金を計算する<br>3. システムが宿泊料金をフロント係に提示する<br>4. フロント係が利用者に料金を案内する<br>5. 利用者が料金を支払う<br>6. システムが支払いを記録する<br>7. システムが予約状態をチェックアウト済みに更新する |
| 代替フロー | 2a. 追加サービスの利用履歴がある場合<br>　2a1. システムが追加料金を宿泊料金に加算する<br>　2a2. 基本フローの手順3に進む |
| 例外フロー | 5a. 利用者が提示された料金で支払いを拒否する場合<br>　5a1. フロント係が利用者と料金内容を再確認する<br>　5a2. 解決しない場合、ユースケースを終了する（チェックアウト未完了） |

## UC4: 予約内容を確認する

| 項目 | 内容 |
|---|---|
| アクター | 利用者、フロント係 |
| 概要 | 利用者またはフロント係が予約番号などを指定し、予約内容を確認する |
| 事前条件 | 予約番号、氏名、宿泊日のいずれかの検索条件を把握している |
| 事後条件 | 該当する予約内容が確認されている |
| 基本フロー | 1. アクターが予約番号を指定する<br>2. システムが予約を検索する<br>3. システムが宿泊日・客室タイプ・予約状態を提示する<br>4. アクターが予約内容を確認する |
| 代替フロー | 1a. アクターが予約番号を指定できない場合<br>　1a1. アクターが氏名・宿泊日を指定する<br>　1a2. 基本フローの手順2に進む |
| 例外フロー | 2a. システムが該当する予約を検出できない場合<br>　2a1. システムが「予約なし」を通知する<br>　2a2. ユースケースを終了する |

## UC5: 予約をキャンセルする

| 項目 | 内容 |
|---|---|
| アクター | 利用者 |
| 概要 | 利用者が予約内容を確認したうえで、予約をキャンセルする |
| 事前条件 | 予約が「予約済み」状態で存在する |
| 事後条件 | 予約が「キャンセル済み」状態に更新されている |
| 基本フロー | 1. 利用者が予約番号を指定する<br>2. システムが予約内容を確認する（include UC4）<br>3. システムが予約内容を利用者に提示する<br>4. 利用者がキャンセルを要求する<br>5. システムが予約状態をキャンセル済みに更新する<br>6. システムがキャンセル完了を利用者に通知する |
| 代替フロー | 4a. キャンセル料が発生する場合<br>　4a1. システムがキャンセル料を利用者に提示する<br>　4a2. 利用者がキャンセル料に同意する<br>　4a3. 基本フローの手順5に進む |
| 例外フロー | 2a. システムが該当する予約を検出できない場合<br>　2a1. システムが「予約なし」を利用者に通知する<br>　2a2. ユースケースを終了する<br>4b. 予約がすでにチェックイン済みまたはキャンセル済みの場合<br>　4b1. システムが「キャンセル不可」を利用者に通知する<br>　4b2. ユースケースを終了する |
